# Refactor: Unify Filter/Renderer/Projects with Derived Tag Registry

## Problem

The concept of "project tags" is owned by three separate sources with no shared source of truth:

- **projects.js** defines what tags each project has (`tags: ['python', 'etl']`)
- **index.html** and **projects.html** define which tags are filterable via duplicate button markup (`<button data-filter="python">`)
- **filter.js** reads both at runtime and assumes they match

This creates two categories of silent failure:

1. **Orphan filter buttons** — a filter button for a tag no project has shows zero results with no error.
2. **Unfilterable projects** — the "Portfolio Website" project has tags `['html', 'css', 'javascript']` but no filter buttons exist for those tags. It's only visible via "All Projects."

Additionally, filter button markup is **duplicated** across both HTML pages (~30 lines each), requiring manual synchronization when tags change.

**Dependency category:** In-process — pure computation, no I/O. All three modules are in-memory JS with no external dependencies. The coupling is entirely through shared string conventions.

## Proposed Interface

Design A: Minimal — derive tags from `projects.js`, making orphan buttons structurally impossible.

### New export in `projects.js`

```js
/**
 * Sorted, deduplicated tags derived from all projects.
 * @type {string[]}
 */
export const tags = [...new Set(projects.flatMap((p) => p.tags))].sort();
```

One line. The tag registry is derived, not maintained. It cannot drift from the projects array because it IS the projects array.

### Display name mapping (for non-obvious labels)

```js
/**
 * Display labels for tags that need non-obvious formatting.
 * Tags not listed here use default formatting (capitalize, split on hyphens).
 * @type {Record<string, string>}
 */
export const TAG_LABELS = {
  etl: 'ETL/ELT',
  r: 'R',
  sql: 'SQL',
  html: 'HTML',
  css: 'CSS',
  'machine-learning': 'Machine Learning',
  'statistical-analysis': 'Statistical Analysis',
  'analytics-dashboard': 'Analytics Dashboard',
  'data-pipelines': 'Data Pipelines',
  'business-intelligence': 'Business Intelligence',
};
```

### New function in `renderer.js`

```js
/**
 * Generates filter buttons from a tag list into a container element.
 *
 * @param {HTMLElement} container - The DOM element to render buttons into
 * @param {string[]} tags - Array of tag IDs to create buttons for
 * @param {Object} [options]
 * @param {string} [options.resetLabel='All Projects'] - Label for the reset button
 * @param {string} [options.resetFilter='all'] - Filter value for the reset button
 */
export function renderFilterButtons(container, tags, options = {});
```

### Optional validation in `initFilter`

```js
// config gains one optional field:
//   config.knownTags: string[] — if provided, warns on unknown data-filter values in DOM
export function initFilter(config = {});
```

### What changes in HTML

Both `index.html` and `projects.html` replace their ~30 lines of filter button markup with an empty container:

```html
<div id="filter-buttons"></div>
```

The buttons are generated by `renderFilterButtons` in the JS initialization flow.

### Usage — Production (main.js)

```js
import { projects, tags } from './projects.js';
import { renderProjectCards, renderFilterButtons } from './renderer.js';
import { initFilter, getFilterFromURL } from './filter.js';

const page = document.body.dataset.page;
const filterContainer = document.getElementById('filter-buttons');
const projectContainer = document.getElementById('projects-container');

renderProjectCards(projectContainer, projects);

if (page === 'projects') {
  renderFilterButtons(filterContainer, tags, { resetLabel: 'All Projects', resetFilter: 'all' });
  initFilter({ maxVisible: null, knownTags: tags, initialFilter: getFilterFromURL() });
} else {
  renderFilterButtons(filterContainer, tags, { resetLabel: 'Featured', resetFilter: 'featured' });
  initFilter({ maxVisible: 4, defaultFilter: 'featured', knownTags: tags });
}
```

### Usage — Tests (no DOM for tag validation)

```js
import { projects, tags } from '../WebContent/js/projects.js';

test('every project tag appears in the derived registry', () => {
  for (const p of projects) {
    for (const t of p.tags) {
      expect(tags).toContain(t);
    }
  }
});

// This test is tautological by design — drift is structurally impossible.
// It documents the invariant for future developers.

test('tags are sorted and unique', () => {
  expect(tags).toEqual([...new Set(tags)].sort());
});
```

```js
// DOM tests for renderFilterButtons
test('renders one button per tag plus reset', () => {
  const container = document.createElement('div');
  renderFilterButtons(container, ['python', 'sql'], { resetLabel: 'All' });
  const buttons = container.querySelectorAll('[data-filter]');
  expect(buttons.length).toBe(3); // reset + python + sql
});
```

### What It Hides Internally

| Hidden inside                                                         | Exposed to callers                              |
| --------------------------------------------------------------------- | ----------------------------------------------- |
| Tag deduplication and sorting logic                                   | `tags` constant (derived, immutable)            |
| Button DOM structure (`<button class="skill-tag" data-filter="...">`) | `renderFilterButtons(container, tags, options)` |
| Display name formatting (heuristic + lookup)                          | `TAG_LABELS` for overrides only                 |
| "All"/"Featured" pseudo-filter button creation                        | `options.resetLabel` / `options.resetFilter`    |
| Button ordering and active-state class assignment                     | Internal to `renderFilterButtons`               |

## Dependency Strategy

**In-process** — pure computation, no I/O, no external dependencies.

| Concern              | How it works                                                         |
| -------------------- | -------------------------------------------------------------------- |
| Tag source of truth  | Derived from `projects` array at module evaluation time              |
| Filter buttons       | Generated by `renderFilterButtons` from the derived `tags`           |
| Tag-to-card matching | Unchanged — `renderer.js` writes `data-tags`, `filter.js` reads them |
| Display names        | `TAG_LABELS` lookup with capitalize-and-split fallback               |

The key structural guarantee: **because both filter buttons and card tags are derived from the same `projects` array, orphan buttons are impossible by construction.**

## Implementation Recommendations

### What the module should own

- `projects.js`: Project data + derived tag list + display name overrides
- `renderer.js`: Card creation + filter button generation (both write to DOM)
- `filter.js`: Filter logic + event wiring (reads from DOM)

### What it should hide

- Tag deduplication/sorting (internal to `tags` derivation)
- Button DOM structure and ordering (internal to `renderFilterButtons`)
- Display name formatting heuristics (internal to `renderFilterButtons`)

### What it should expose

- `tags` — derived, sorted, deduplicated tag list
- `TAG_LABELS` — display name overrides for non-obvious tags
- `renderFilterButtons(container, tags, options)` — button generator
- `initFilter(config)` — unchanged, with optional `knownTags` validation

### How callers should migrate

1. Add `tags` and `TAG_LABELS` exports to `projects.js` (one line + one object)
2. Add `renderFilterButtons` function to `renderer.js` (~15-20 lines)
3. Add optional `knownTags` warning to `initFilter` (~3-5 lines)
4. Update `main.js` to call `renderFilterButtons` before `initFilter`
5. Delete duplicate filter button markup from `index.html` (~30 lines) and `projects.html` (~30 lines)
6. Replace with empty `<div id="filter-buttons"></div>` container in each HTML file
7. Add tag validation tests to `__tests__/projects.test.js`
8. Add `renderFilterButtons` tests to `__tests__/renderer.test.js`
